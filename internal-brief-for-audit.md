# WIRE — внутренний бриф проекта (для внешнего аудита)

> Этот документ — полное описание проекта: что есть сейчас, что строим, как это устроено
> технически и продуктово, экономика и риски.
> **Задача аудитора:** прочитать целиком и дать критический разбор — см. раздел 12 с вопросами.

---

## 1. Контекст

Команда ведёт новостные Telegram-каналы (тематика: Украина/Россия, война, регионы; планируется
расширение на крипту, финансы, спорт, технологии). Есть работающая самодельная система
агрегации, которую хотим превратить в платформу: сначала для себя, затем — как сервис
для других каналов по подписке. Разработку и эксплуатацию ведёт один техлид.

## 2. Текущая система (as-is, работает в проде)

```
576+ ТГ-каналов ──> userbot (Python/Telethon, подписан на каналы, push-события)
                        │ пересылает всё в служебного бота
                        ▼
                 node-бот (Telegraf, JS)
                        │ маршрутизирует по хардкод-спискам 1channels.js…4channels.js
                        ▼
        общий канал + 4 тематических канала + рассылка юзерам

отдельно: RSS-бот (парсит новостные сайты: BBC и др.) → свои каналы
```

Особенности и боли:
- Маршруты зашиты в JS-файлах — любое изменение = правка кода + рестарт pm2.
- Фильтр рекламы примитивный (стоп-слова), нативную рекламу не ловит.
- Дубли не склеиваются: одно событие приходит из десятков каналов повторами.
- Ничего не сохраняется (нет БД) — только пересылка.
- Альбомы собираются буфером по `grouped_id` (ожидание ~0.5–2.5 с), медиа не скачиваются
  (используются телеграмовские file_id, которые протухают).
- Юзербот-аккаунт уже один раз теряли (слетели все сессии) — восстанавливали вручную,
  подписка на 400+ каналов заново, ловили FloodWait.
- Сервер: VPS 1 vCPU / 1GB RAM, на нём же ~10 других pm2-процессов, память под завязку.

## 3. Целевая архитектура (to-be)

Принцип: **одно ядро — много выходов.** Всё пишется в базу без фильтрации на входе,
размечается один раз, любой потребитель — фильтр поверх базы.

```
ВХОДЫ                        ЯДРО                            ВЫХОДЫ
1000+ ТГ-каналов (userbot)   PostgreSQL (всё: текст,         наши каналы (маршруты)
RSS-ленты                    entities, медиа→S3/R2,          клиентские боты (подписка)
API первоисточников          views/forwards)                 web-лента + Mini App
(тревоги, биржи) [позже]     AI-разметка (тема/язык/         AI-редактор (approve→публикация)
клиентские каналы (поллинг)  реклама/важность 1–5)           дайджесты
                             дедуп (embeddings+pgvector)
```

Ключевые решения:
- **Сборщики остаются раздельными** (ТГ на Python/Telethon, RSS на Node), объединяются
  на уровне формата записи в БД. Доставщик — один сервис.
- **Запись в базу параллельна текущей пересылке** (упала база — пересылка живёт).
- **Медиа скачиваются сразу** в объектное хранилище (Cloudflare R2): file_id протухают,
  посты удаляют (в военных каналах постоянно), для веба телеграмовские id бесполезны.
- Сохраняются telegram-entities (форматирование) — для рендера 1-в-1 в вебе.
- Новый VPS (Hetzner CX32+), docker-compose, всё в конфиге через .env.

## 4. AI-слой

Три задачи — три разных инструмента (микс провайдеров, переключаемый через конфиг):

| Задача | Объём | Модель | ~$/мес |
|---|---|---|---|
| Разметка: тема (мультилейбл из ~10), язык, реклама да/нет, важность 1–5 | ~150k постов/мес (после пре-фильтра) | дешёвая (Gemini Flash-Lite / GPT-nano класс; fallback Claude Haiku) | $15–30 (Haiku: $80–120) |
| Дедупликация | тот же поток | embeddings (Voyage-3-lite) + pgvector, cosine similarity | $2–5 |
| AI-редактор: отбор кандидатов + рерайт в стиле канала | ~50 постов/день | Claude Sonnet | $15–30 |

Оптимизации: пре-фильтр правилами ДО AI (стоп-слова, хэш-дубли, каналы вне маршрутов —
режет ~50–70% потока бесплатно); дубль размечается один раз (метки наследуются кластером);
Batch API для несрочного (−50%); prompt caching.

**Дедуп-логика:** событие создаётся с первого поста и доставляется мгновенно; последующие
похожие посты (embedding-похожесть выше порога + окно 2–4 часа + совпадение темы)
приклеиваются к событию, обновляя счётчик «⧉ N источников» и «первым сообщил».
Консервативный порог: лучше не склеить, чем склеить разное. Кнопка «расклеить» у оператора.
Склейка считается всегда (для ленты), а в доставке — переключатель клиента
(склеивать / слать все повторы).

**AI-редактор:** читает размеченный поток → отбирает «интересное для канала» (критерии
задаются примерами опубликованных постов) → присылает в админ-канал кандидата с готовым
черновиком в стиле издания (стиль — из 20–50 примеров постов) → кнопки
✅ опубликовать / ✏️ переписать / ❌ пропустить → решения пишутся в базу и подмешиваются
в промпт отбора (память вкуса). Плюс ручной режим: кнопка «Редактировать» на любой новости
открывает студию (правка текста/заголовка/медиа, AI-варианты, публикация в канал/группу/ЛС).

## 5. Стратегия источников и юзерботы

- Лимит подписок аккаунта: 500 (1000 с Premium). Сейчас один аккаунт с Premium.
- **Трёхуровневая схема чтения:**
  1. Основной пул — аккаунты подписаны → мгновенный push (Telethon events).
  2. Клиентские публичные каналы — **поллинг без подписки** (MTProto позволяет читать
     публичные каналы по username не вступая): не тратит лимит, задержка 10–60 с.
  3. Популярные клиентские каналы переводим в подписку (пул растёт за счёт клиентов).
- Приватные каналы клиентов — по приглашению нашего аккаунта-читалки (фича дорогого тарифа, v2).
- Расширение пула до 1000+: авто-кандидаты из `fwd_from` (кого чаще всех репостят наши
  источники), поиск TGStat по категориям, ручной добор топ-каналов тем.
- Отказоустойчивость: 2–3 аккаунта (чтение пула / поллинг / холодный резерв), детектор
  молчащих каналов, детектор пропусков по сквозным message id (дозапрос дыр), алерты.

## 6. Web-платформа и Mini App

Один код (React+TS+Vite) — три поверхности: web, PWA, Telegram Mini App (авторизация
через initData, MainButton/BackButton, тема из themeParams). Mock-first разработка:
фронт строится на MSW по зафиксированному API-контракту (сущности Event / Route / Source / Me,
REST + WebSocket `event.new` / `event.updated`), ядро подключается сменой URL.

Экраны: лента (3 колонки, виртуализация, LIVE через WS, фильтры в URL), страница события
(все источники с временами), маршруты (визард 3 шага: темы+фильтры → куда → формат:
оригинал/карточка/дайджест), источники (каталог пула + добавление своего с AI-превью
и антиспам-проверкой), студия редактора, онбординг ≤60 сек / 3 нажатия,
пульт оператора `/ops` (статусы сервисов, расход AI, здоровье источников, контроль
качества разметки с ручной выборкой 50 постов/нед).

Дизайн-система «WIRE»: тёмный ньюсрум-терминал + светлая тема, янтарный акцент,
токены CSS, 15 базовых компонентов, NewsCard с MediaGrid (альбомы до 10, видео).

## 7. Продукт и монетизация (после внутреннего запуска)

- **Модель Б1 «бот-в-группу»:** клиент добавляет бота, в Mini App выбирает темы/фильтры,
  получает ленту из готового пула. Онбординг целиком в Telegram, без сайта. Плюс свои источники.
- Тарифы (прикидка): Старт $12 (2–3 темы, 1 группа) / Про $29 (все темы, ключевые слова,
  3 группы, web) / Редакция $79 (AI-редактор под стиль клиента).
- Скорость как ось тарифов: дешёвым — задержка пару минут, дорогим — мгновенно
  (техподоплёка: push vs поллинг, у нас почти бесплатно).
- Web-платформа с оплатой картой — этап после проверки спроса ботом.

| Редактор под стиль | — | — | — | частично | **отбор+черновик+approve+память** |
| Цены | €3–18/мес (Business €99+) | $7–15 | $10–16 | 99–350₽ пакеты | $12–79 |

Механика Junction подтверждена: дешёвые тарифы — их аккаунты читают публичные каналы
поллингом (задержка = тариф), дорогой — клиент подключает свой аккаунт как читалку
(MTProto-сессия на аккаунте клиента). UX-паттерн их Mini App: визард «что хотите → источник →
куда → готово», демо без карты, NPS-опрос. Слабость: интерфейс-простыня 40+ опций,
юзер должен сам знать источники.

## 9. Экономика (операционные расходы)

- Операционка MVP (300 активных каналов, пре-фильтр): AI $35–65 + сервер $15–30 +
  R2 $5–20 + премиум-аккаунты ~$10 ≈ **$70–125/мес**. Растёт с объёмом.
- Потолок без оптимизаций на 1000+ каналов «в лоб»: $300–800/мес только AI (не наш режим).
- Окупаемость при продаже: 3–5 подписчиков Про покрывают операционку.

## 10. Риски (осознаны, план снижения)

1. **Юзербот-аккаунты** — единая точка отказа, банят волнами; уже теряли. → мультиаккаунт,
   failover, процедура восстановления, поллинг как резервный канал чтения.
2. **Качество дедупа** — ложные склейки (два обстрела в один день) хуже недосклейки. →
   консервативные пороги, окно времени, совпадение темы, ручное расклеивание.
3. **Качество разметки** — >10–15% ошибок тем убивают доверие. → еженедельная ручная
   выборка, метрика точности в пульте, исправления → few-shot примеры в промпт.
4. **Правовая серость** — перепродажа чужого контента. → всегда ссылка на оригинал
   (атрибуция + трафик каналам), в платном API — выжимка+ссылка, не полный текст.
5. **Bus-фактор = 1** (один техлид). → пульт, алерты в ТГ, документация, docker-compose.
6. **ToS AI-провайдеров** — движок продукта строго на API-ключах (не на потребительских
   подписках), микс провайдеров через конфиг.

## 11. Дорожная карта (очерёдность, без дат)

1. Новый VPS + PostgreSQL + запись всего в базу (параллельно текущей пересылке) + медиа в R2
2. AI-разметка (пре-фильтр → дешёвая модель) + дедуп (embeddings) + метки в базе
3. Управление: бот + Mini App (маршруты, фильтры, форматы) — замена хардкод-спискам
4. AI-редактор MVP (отбор + черновик + approve) + студия «Редактировать»
5. Web-лента + кабинет (тот же код) + пульт оператора
6. Расширение пула до 1000+ (скаут по репостам, TGStat, новые темы) + второй аккаунт
7. Продажа: онбординг, тарифы, оплата (Stars/карта) — после приёмки 1–5

## 12. Вопросы к аудитору

Дай критический разбор по пунктам, без комплиментов, с конкретикой:

1. **Архитектура:** есть ли дыры в схеме «одно ядро — много выходов»? Что сломается первым
   на объёме 40k постов/день? Достаточно ли PostgreSQL+pgvector или нужен отдельный
   поисковый/векторный движок?
2. **Дедуп:** оцени выбранный подход (embeddings + порог + окно + тема). Какие known-failure
   режимы мы не учли? Нужен ли LLM-судья для пограничных пар?
3. **AI-экономика:** реалистичны ли оценки $35–65/мес? Где мы недооцениваем токены
   (длинные посты, медиа-описания, ретраи)?
4. **Юзерботы:** насколько велик риск бана при поллинге сотен публичных каналов с 2–3
   аккаунтов? Лучшие практики анти-бана, которые мы не назвали?
5. **Продукт:** правдоподобна ли ценность «готовый пул + склейка + редактор» против
   Junction/TeleFeed? Какой сегмент заплатит первым? Что выкинуть из MVP как лишнее?
6. **Тарифы:** $12/29/79 — адекватно? Скорость как ось тарифа — рабочая идея или усложнение?
7. **Юридика:** оцени риск перепродажи агрегированного контента ТГ-каналов (с атрибуцией)
   для платного сервиса. Прецеденты, о которых стоит знать?
8. **Фронт:** mock-first + один код на web/Mini App — подводные камни? Чего не хватает
   в API-контракте (Event/Route/Source) под будущие фичи (дайджесты, API-доступ, white-label)?
9. **Чего вообще нет в плане**, а должно быть? (безопасность, бэкапы, GDPR, модерация
   контента, abuse клиентских источников и т.п.)
10. Если бы ты строил это сам — что бы сделал принципиально иначе?
