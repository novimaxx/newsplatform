# WIRE — задание на рабочий прототип (Web + Telegram Mini App)

> **Цель:** собрать РАБОТАЮЩИЙ кликабельный прототип фронта на фейковых данных.
> Запускается одной командой, выглядит как готовый продукт, все экраны кликаются.
> Бэкенда нет — все данные из моков. Позже реальное API подключается **сменой одного слоя**,
> без переписывания экранов.
>
> **Что НЕ делаем сейчас:** реальный сбор новостей, базу, AI, юзерботов. Только интерфейс.
> **Что проверяем:** как это выглядит и ощущается, поток экранов, дизайн, UX Mini App.
>
> Дизайн-референс: `news-platform-plan.html`. Полная спека фронта: `web-platform-spec.md`
> (оттуда — токены, компоненты, детали экранов). Этот документ — как собрать демо-версию.

---

## 1. Что должно получиться

Одно React-приложение, которое открывается:
- **в браузере** — `http://localhost:5173` → полноценный сайт с лентой и кабинетом;
- **как Telegram Mini App** — через бота (или тестовый режим `?tgWebApp=1`) → те же экраны,
  адаптированные под телефон, с нативными кнопками TG.

Всё наполнено **реалистичными фейковыми новостями** (война, крипта, мир, финансы — как в
настоящих каналах), чтобы демо выглядело живым, а не «Lorem ipsum».

---

## 2. Стек (минимальный, чтобы просто запускалось)

```
React 18 + TypeScript + Vite
React Router v7        — экраны
Zustand               — состояние (фильтры, выбранный маршрут, тема)
CSS-переменные + CSS Modules — дизайн-токены
Lucide React          — иконки
Day.js                — «43 сек назад»
@telegram-apps/sdk    — Mini App (initData-заглушка, MainButton, тема)
```

Никаких бэкенд-зависимостей, никакой БД. Данные — из `src/mocks/`.
Запуск: `npm install && npm run dev`.

---

## 3. Архитектура «моки сейчас — API потом»

Ключевое требование. Все данные идут через **слой доступа**, а не напрямую в компонентах:

```ts
// src/data/index.ts — единственная точка, которую потом переключим на реальное API
export const dataSource: DataSource =
  import.meta.env.VITE_USE_MOCKS === 'true'
    ? mockDataSource      // сейчас: возвращает фейковые данные с искусственной задержкой
    : apiDataSource;      // потом: реальные fetch к /api/v1 (заглушка, не реализуем сейчас)

interface DataSource {
  getFeed(filters): Promise<Event[]>;
  getEvent(id): Promise<Event>;
  getRoutes(): Promise<Route[]>;
  createRoute(data): Promise<Route>;
  updateRoute(id, patch): Promise<Route>;
  getSources(scope, query): Promise<Source[]>;
  resolveSource(url): Promise<SourcePreview>;
  getMe(): Promise<Me>;
  // ops
  getOpsStatus(): Promise<OpsStatus>;
  getOpsMetrics(): Promise<OpsMetrics>;
}
```

Компоненты вызывают `dataSource.getFeed(...)` и не знают, моки это или сервер.
Когда появится бэкенд — пишем `apiDataSource` и меняем `.env`. Экраны не трогаем.

Типы `Event / Route / Source / Me` — **точно те же**, что в `web-platform-spec.md` раздел 6.2
(это будущий контракт с ядром, менять нельзя).

**Live-имитация:** `mockDataSource` раз в 8–15 сек «вбрасывает» новое событие в ленту
сверху (с подсветкой), чтобы демо ощущалось живым и LIVE-бейдж был не декорацией.

## 4. Фейковые данные (чтобы выглядело реально)

`src/mocks/events.ts` — 30–40 готовых событий, реалистичных:
- разные темы (война, крипта, мир, финансы, спорт) с правильными чипами;
- разная важность (1–5), несколько с `is_ad:true` (показать скрытие рекламы);
- склейка: у части событий `sources` из 5–40 каналов, «первым сообщил @…», разное время;
- медиа: часть с 1 фото, часть с альбомом 4 фото, часть с видео (заглушки-градиенты
  или локальные картинки в `public/mock/`), часть только текст;
- реальные на вид заголовки и первые строки (RU/UK), время «минуты назад».

`src/mocks/routes.ts` — 3 маршрута (канал/группа/ЛС, разные форматы, вкл/пауза, статистика).
`src/mocks/sources.ts` — ~60 каналов для каталога + 12 «моих» (разные статусы, включая
`rejected` со спам-причиной).
`src/mocks/me.ts` — юзер с ролью admin (чтобы был доступ к пульту) и планом.

Данные вынести отдельно — потом легко удалить, когда придёт реальное API.

## 5. Экраны (что показываем в прототипе)

Все из `web-platform-spec.md` раздел 4. Приоритет и что интерактивно:

| Экран | Маршрут | В прототипе |
|---|---|---|
| **Лента** | `/feed` | ⭐ главный. Фильтры реально фильтруют моки, live-вброс, карточки кликаются |
| Событие | `/feed/:id` | полный текст, все источники с временами, кнопки (заглушки) |
| **Маршруты** | `/routes` | список, тумблер вкл/пауза работает (локально), удаление с confirm |
| Визард маршрута | `/routes/new` | 3 шага реально проходятся, в конце «создаётся» (в моках) |
| **Источники** | `/sources` | каталог с поиском/фильтром; добавление своего → фейк-превью канала |
| **Студия** | `/editor/:id` | layout, AI-кнопки показывают «loading» и подставляют готовый текст из мока |
| **Онбординг** | `/onboarding` | ⭐ для Mini App: 3 шага темы→куда→формат, ≤60 сек |
| Настройки | `/settings` | тема (свитч работает!), язык, план (показ) |
| **Пульт** | `/ops` | ⭐ дашборд с фейковыми, но живыми цифрами (тикают), статусы, здоровье источников |

⭐ = обязательно должно выглядеть отполированно (это то, что показываем).

**Интерактив, который реально работает (локально, без бэка):**
- переключение тем/фильтров в ленте → лента меняется;
- тумблеры маршрутов, свитч темы light/dark;
- проход визарда и онбординга по шагам;
- поиск в каталоге источников;
- AI-кнопки в студии (имитация задержки → подстановка готового результата);
- live-вброс новых событий в ленту.

**Заглушки (кнопка есть, но действие — тост «в демо недоступно»):**
- реальная публикация, реальное «добавить бота в группу», оплата.

## 6. Telegram Mini App — что показать

- Определение среды: если открыто в TG (`window.Telegram.WebApp`) или флаг `?tgWebApp=1` —
  включается MiniAppShell (мобильный layout, нативные кнопки).
- **MainButton** — главное действие экрана (визард: «Далее»/«Создать»; студия: «Опубликовать»).
- **BackButton** — назад по роутеру.
- **Тема** — берётся из `themeParams` (тёмный клиент → тёмная тема); в браузере — свитч.
- **Haptic** — вибро на успех/переход (в браузере — no-op).
- Онбординг открывается первым при «первом входе» (флаг в localStorage).
- Тестировать можно: локальный туннель (ngrok/cloudflared) → BotFather → кнопка Mini App,
  либо просто `?tgWebApp=1` в браузере для проверки мобильного вида.

## 7. Дизайн

Взять **целиком** из `web-platform-spec.md` раздел 3 (токены, темы, компоненты) и
`news-platform-plan.html` (визуальный референс). Кратко:
- тёмная тема по умолчанию + светлая равноценная, переключатель;
- янтарный акцент, ньюсрум-стиль, пульс-лого, LIVE-бейдж;
- NewsCard с MediaGrid (1/4 фото, видео), чипы тем по цветам, полоски важности;
- все состояния: skeleton при загрузке моков (с искусственной задержкой 300–600 мс), empty, error.

Собрать UI-kit (`src/ui/`) из раздела 3.3 спеки — компоненты переиспользуются везде.

## 8. Структура

```
src/
├─ app/        # роутер, WebShell / MiniAppShell, провайдер темы
├─ ui/         # дизайн-система (Button, Chip, Switch, NewsCard, MediaGrid, …)
├─ features/   # feed, routes, sources, editor, onboarding, settings, ops

public/mock/   # картинки-заглушки для медиа
```

## 9. Порядок сборки

1. Vite + токены + темы + переключатель light/dark (проверить: тема меняется везде).
2. UI-kit: Button, Chip, Switch, Card, NewsCard, MediaGrid, ImportanceBars, SourceBadge, Skeleton.
3. Моки + DataSource (mock-реализация с задержкой и live-вбросом).
4. **Лента** `/feed` — 3 колонки (web) / 1 (мобайл), фильтры работают, карточки, live.
5. Событие `/feed/:id`.
6. **Маршруты** + визард (он же онбординг).
7. **Источники** — каталог + добавление своего с фейк-превью.
8. **Студия** редактора.
9. **MiniAppShell** — мобильный layout, MainButton/BackButton, тема из TG, `?tgWebApp=1`.
10. **Пульт** `/ops` — живые фейковые метрики.
11. Настройки, пустые/error состояния, полировка.
