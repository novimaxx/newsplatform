## 1. Проект в двух абзацах

Команда ведёт новостные Telegram-каналы. Есть работающая самодельная система: юзербот
(Python/Telethon) читает 400+ каналов и пересылает в бота-маршрутизатор (Node/Telegraf),
который по хардкод-спискам раскидывает по своим каналам. Плюс отдельный RSS-парсер сайтов.
Всё это — «тупая труба»: пересылает всё подряд, дубли не склеивает, реклама фильтруется слабо,
маршруты меняются только правкой кода, ничего не сохраняется.

**Что строим (WIRE):** превращаем трубу в платформу. Одно ядро — база (PostgreSQL) хранит
всё, AI размечает по темам/важности/рекламе и склеивает дубли в события; поверх — три поверхности
одного продукта (Telegram Mini App, Web/PWA, Admin-панель), доставка по настраиваемым маршрутам,
и AI-редактор для собственных изданий. Сначала для себя, потом — как сервис по подписке.

Кодовое имя / бренд в макетах: **WIRE** (рабочая заглушка, финальное имя не выбрано).

## 2. Что работает сейчас (as-is)

- `userbot.py` — юзербот на Telethon: читает каналы из `channels.json`, пересылает боту,
  собирает альбомы, помечает прочитанным. Работает в проде на сервере.
- Node-бот (`my_node_bot`, не в этой папке) — принимает пересылки, раскидывает по каналам
  A/B/C/D по спискам `1channels.js`…`4channels.js`, рассылает юзерам.
- RSS-боты — парсят новостные сайты.
- Сервер: Hetzner-подобный VPS, всё под pm2. Текущий узкий: 1 vCPU / 1GB (нужен новый под базу+AI).
- Вспомогательные скрипты: `auth.py`, `subscribe_all.py`, `mute_all.py`, `update_channels.py`,
  `list_channels.py`, `export_html.py`.
- Боли: маршруты в коде, слабый фильтр рекламы, нет склейки дублей, нет базы, file_id медиа
  протухают, юзербот-аккаунт уже теряли.

## 3. Зафиксированные решения (лог)

Эти решения приняты и лежат в основе спеки. Менять — только осознанно и синхронно во всех доках.

**Продукт / MVP:**
- MVP = уже полноценная первая версия ВСЕЙ системы (не урезанная): три поверхности
  (Mini App + Web/PWA + Admin), лента, маршруты, AI-разметка, склейка, AI-редактор,
  основа под платных пользователей. Web и кабинет НЕ выносим «на потом».
- Бот подключается **и в группу, и в канал** (оба варианта).
- Источники: **готовый пул 1000+ по темам ПЛЮС клиент может добавить свои каналы.**
- Клиент в первой версии — **только авто-маршруты** (настроил темы/фильтры → получает поток).
  Ручной выбор новости из ленты и студия-редактор = ваш внутренний инструмент, не базовый клиент.
- AI-редактор в MVP — **только ваши издания** (2–3 канала на старте), клиентам как топ-тариф
  включим позже (сущности заложены сразу).
- Подписки/тарифы — сущности в MVP сразу (trial, дата окончания, лимиты, ручное назначение
  админом); автооплата — позже.

**Архитектура:**
- Принцип «одно ядро — много выходов». Вход пишет всё без фильтрации, разметка одна на всех.
- Отфильтрованные посты НЕ удаляем — храним с причиной (`status='filtered'`, `filter_reason`,
  `filter_version`), чтобы проверять ошибки фильтра и переприменять правила.
- Медиа скачиваем сразу в объектное хранилище (Cloudflare R2) — file_id протухают, посты удаляют.
- Сохраняем telegram-entities (форматирование) для рендера 1-в-1 в вебе.

**Telegram-аккаунты (важно):**
- Аккаунт = **набор назначенных источников, НЕ тема.** Тема вешается на пост после получения.
- 4 активных сборщика (200–300 источников каждый) + 1 резерв; критичные источники дублируются
  на 2 аккаунтах; распределение по нагрузке/FloodWait/priority; failover на резерв;
  перенос источника между аккаунтами сменой поля в БД, без кода.

**AI:**
- Микс провайдеров, переключаемый через `.env`: дешёвая модель на разметке (Gemini Flash-Lite /
  GPT-nano класс, fallback Haiku), embeddings на дубли (Voyage), Claude Sonnet на редактор.
- Пре-фильтр правилами ДО AI (дёшево срезает 50–70%), AI — где правила не справляются.
- Дедуп консервативный (порог похожести + окно 2–4 ч + совпадение темы), лучше не склеить,
  чем склеить разное; merge/unmerge вручную в админке.
- Движок продукта — ТОЛЬКО на API-ключах, НЕ на потребительских подписках (Max/ChatGPT/Codex):
  подписки для интерактивной разработки, API — для автоматического бэкенда (лимиты + ToS).

**Экономика (операционка MVP, ориентир):** AI $35–65 + сервер $15–30 + R2 $5–20 ≈ $70–125/мес
(300 активных каналов с пре-фильтром); растёт с объёмом.

## 4. Целевая архитектура (кратко)

```
ВХОДЫ                      ЯДРО                         ВЫХОДЫ
tg-collector ×4 (Telethon) PostgreSQL + pgvector        router → маршруты клиентов
tg-poller (публичные без   media → R2                   editor-bot → ваши издания
  подписки)                queue (Redis/BullMQ)         web-api → Web + Mini App + Admin
rss-collector              ai-workers:
                            tagger (разметка)
                            deduper (embeddings)
                            editor-agent (рерайт)
```
Детали, схема БД, контракты, промпты, порядок сборки — в `BUILD-SPEC.md`.

## 6. Состав MVP (согласованный)

1. **Сбор:** 4+1 аккаунтов, 1000–1500 источников, управление распределением, RSS, медиа в R2,
   восстановление пропусков.
2. **Обработка:** существующие фильтры (с сохранением причины) → AI (тема/важность/реклама/язык)
   → точные и смысловые дубли → события с несколькими источниками.
3. **Продукт для пользователя:** Telegram-бот, Mini App, Web/PWA, авторизация через Telegram,
   лента, фильтры, источники, маршруты, настройки подписки.
4. **Admin-панель:** пользователи, тарифы/подписки, маршруты юзера, подключённые чаты,
   аккаунты-сборщики, источники и распределение, состояние конвейера, расходы AI, ошибки
   доставки, ручная проверка разметки, merge/unmerge, блокировки, журнал действий.
5. **AI-редактор:** очередь кандидатов, AI-черновик, редактирование текста/медиа, выбор канала,
   публикация и отложка, история решений (2–3 ваших издания).
6. **Продажи (сущности):** тариф в системе, ручное назначение подписки, лимиты, trial,
   дата окончания, блокировка после окончания. Автооплата — позже.
